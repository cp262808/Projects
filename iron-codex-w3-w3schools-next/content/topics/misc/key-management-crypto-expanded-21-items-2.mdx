<summary>Key Management → KMS/HSM Lifecycle</summary>
<p class="tags"><b>Tags:</b> KMS, HSM, Rotation</p>
<p><b>Question:</b> Are keys generated in FIPS 140-2/3 modules and rotated ≤ 12 months with auditable usage?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SC-12, SC-13<br/>NIST SP 800-57<br/>- ISO 27001: A.10.1.2</p>
<p><b>Applicability:</b> All encrypted data</p>
<p><b>Expected Result:</b> CMKs never leave HSM; periodic rotation; usage logs; dual control for deletes.</p>
<p><b>Why It Matters:</b> Key theft nullifies encryption.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Split KMS admin vs crypto user roles; approvals for schedule deletions.</li>
<li>Envelope encryption with frequent DEK rotation.</li>
</ol>

<summary>Key Management → Envelope Encryption</summary>
<p class="tags"><b>Tags:</b> Envelope, AEAD</p>
<p><b>Question:</b> Do apps use envelope encryption (per-object DEK wrapped by CMK)?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SC-12</p>
<p><b>Applicability:</b> Databases/files/PII blobs</p>
<p><b>Expected Result:</b> Random DEK per item; stored as wrapped blob; cache DEKs in memory only.</p>
<p><b>Why It Matters:</b> Limits blast radius if DEK exposed.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Use AEAD (AES-GCM/ChaCha20-Poly1305) with unique nonces.</li>
<li>Never log plaintext keys; wipe memory buffers.</li>
</ol>

<ol>
<li>Use KMS grants; deny human Decrypt; require approvals.</li>

<summary>Key Management → KEK/DEK Rotation Cadence</summary>
<p class="tags"><b>Tags:</b> Rotation Policy</p>

<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SC-12</p>
<p><b>Applicability:</b> Encrypted stores and backups</p>

<p><b>Why It Matters:</b> Balances performance with risk.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Stagger rotations; monitor for key exhaustion; document playbooks.</li>
