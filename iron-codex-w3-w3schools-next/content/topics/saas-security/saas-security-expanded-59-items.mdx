<summary>SaaS Security → Tenant Isolation</summary>
<p class="tags"><b>Tags:</b> Multi-tenant, RLS</p>
<p><b>Question:</b> Does the SaaS enforce tenant isolation at data/execution with per-tenant keying?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SC-4, SC-7, SC-28<br/>- ISO 27001: A.14.2.7<br/>- SOC 2: CC6.6</p>
<p><b>Applicability:</b> Multi-tenant SaaS</p>
<p><b>Expected Result:</b> Tenant ID validation on every access; RLS; per-tenant keys or strict grants.</p>
<p><b>Why It Matters:</b> Prevents cross-tenant leaks.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Middleware enforces tenant claims; block cross-tenant joins.</li>
<li>Pen tests include cross-tenant attempts.</li>
</ol>

<summary>SaaS Security → SSPM &amp; Config Drift</summary>
<p class="tags"><b>Tags:</b> SSPM, Drift</p>
<p><b>Question:</b> Is SaaS posture monitored for misconfig and drift?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: CA-7, RA-5</p>
<p><b>Applicability:</b> Core SaaS (M365, Google, Slack, Salesforce)</p>
<p><b>Expected Result:</b> Continuous checks; alert on public shares/legacy auth/weak MFA.</p>
<p><b>Why It Matters:</b> Early detection of risky settings.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Baseline desired config; ticket on drift; export to SIEM.</li>
</ol>

<summary>SaaS Security → OAuth App Governance</summary>
<p class="tags"><b>Tags:</b> OAuth, CASB</p>
<p><b>Question:</b> Do you restrict third-party OAuth apps and review scopes/consent?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: AC-19, SI-4</p>
<p><b>Applicability:</b> Google/M365/Slack/Salesforce</p>
<p><b>Expected Result:</b> Admin consent workflows; block risky scopes; review tokens.</p>
<p><b>Why It Matters:</b> SaaS-to-SaaS exfil path.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Limit offline_access; alert on new app installs; remove dormant apps.</li>
</ol>
