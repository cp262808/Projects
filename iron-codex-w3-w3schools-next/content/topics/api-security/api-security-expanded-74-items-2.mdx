<summary>API Security → OAuth Scope Restriction</summary>
<p class="tags"><b>Tags:</b> OAuth, OIDC, Least Privilege</p>
<p><b>Question:</b> Are OAuth access tokens issued with minimum necessary scopes/audiences per client and API resource?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: AC-6, IA-2(1), SC-23<br/>- ISO 27001: A.9.4.1, A.14.1.2<br/>- SOC 2: CC6.1, CC6.2<br/>- OWASP API Top 10: API5, API6</p>
<p><b>Applicability:</b> Any API using OAuth 2.0/OIDC (SaaS or custom).</p>
<p><b>Expected Result:</b> Short-lived tokens (≤ 1h), least-privilege scopes, correct 'aud'. Refresh tokens rotated/bounded (≤ 30d).</p>
<p><b>Why It Matters:</b> Overscoped/long-lived tokens increase blast radius and replay risk.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Define per-resource scopes; avoid wildcards.</li>
<li>Authorizer validates 'scope' and 'aud' every request.</li>
<li>Use PKCE for public clients; revoke on compromise.</li>
</ol>
<p><b>Example Config / Command:</b></p>
<div class="copywrap"><pre># JWT scope/aud check (pseudo)
assert 'read:reports' in claims['scope'].split()
assert claims['aud'] == 'https://api.example.com'
</pre><button class="copybtn" type="button">Copy</button></div>

<summary>API Security → mTLS Between Services</summary>
<p class="tags"><b>Tags:</b> mTLS, Service Mesh, Zero Trust</p>
<p><b>Question:</b> Is mutual TLS enforced for service-to-service calls, with automated certificate rotation?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SC-8, SC-12, SC-13<br/>- ISO 27001: A.13.2.3<br/>- SOC 2: CC6.6</p>
<p><b>Applicability:</b> Microservices/internal APIs (K8s/service mesh).</p>
<p><b>Expected Result:</b> STRICT mTLS at gateway/sidecar; cert rotation ≤ 90d; private CA; SAN/SPIFEE validated.</p>
<p><b>Why It Matters:</b> Prevents impersonation and on-path attack on east-west traffic.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Adopt Istio/Linkerd or gateway mTLS policies.</li>
<li>Use SPIFFE/SPIRE identity issuance; auto-rotate.</li>
<li>Pin expected SAN/SPIFFE per policy.</li>
</ol>
<p><b>Example Config / Command:</b></p>
<div class="copywrap"><pre>apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
spec:
  mtls:
    mode: STRICT
</pre><button class="copybtn" type="button">Copy</button></div>

<summary>API Security → Rate Limiting &amp; Abuse Controls</summary>
<p class="tags"><b>Tags:</b> DoS, Bots, Quotas</p>
<p><b>Question:</b> Do you enforce per-user/IP/token rate limits with adaptive throttling and circuit breakers?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SI-4, SC-5<br/>- ISO 27001: A.12.1.3<br/>- SOC 2: CC7.1<br/>- OWASP API Top 10: API4</p>
<p><b>Applicability:</b> Public/partner APIs; auth and costly operations.</p>
<p><b>Expected Result:</b> Layered limits; 429 w/ Retry-After; WAF bot mitigation; anomaly alerts.</p>
<p><b>Why It Matters:</b> Mitigates brute force, credential stuffing, and resource exhaustion.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Define per-plan quotas; stricter for /auth/* and writes.</li>
<li>Sliding windows + bursts; SIEM alerts on abuse.</li>
<li>Backoff clients; block abusive tokens.</li>
</ol>
<p><b>Example Config / Command:</b></p>
<div class="copywrap"><pre># Kong rate limit (declarative)

- name: rate-limiting
  config: { minute: 100, policy: redis }

<summary>API Security → Schema &amp; Payload Validation</summary>
<p class="tags"><b>Tags:</b> OpenAPI, Validation</p>
<p><b>Question:</b> Are bodies validated against versioned schemas with strict types and bounds?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SI-10, SA-11<br/>- ISO 27001: A.14.2.5<br/>- SOC 2: CC7.2<br/>- OWASP API Top 10: API8</p>
<p><b>Applicability:</b> All JSON/XML/GraphQL APIs.</p>
<p><b>Expected Result:</b> Validation at gateway/service; reject unknown fields; max size; numeric bounds; enum allowlists.</p>
<p><b>Why It Matters:</b> Prevents injection, mass assignment, deserialization attacks.</p>

</ol>

<summary>API Security → BOLA/BFLA Authorization</summary>
<p class="tags"><b>Tags:</b> IDOR, AuthZ, Multi-tenant</p>
<p><b>Question:</b> Do you enforce object/function-level authorization on every call (server-side)?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: AC-3, AC-6<br/>- ISO 27001: A.9.1.2, A.9.4.1<br/>- SOC 2: CC6.6<br/>- OWASP API Top 10: API1, API5</p>
<p><b>Applicability:</b> Multi-tenant APIs and any resource with ownership.</p>
<p><b>Expected Result:</b> Per-request ABAC/RBAC; deny cross-tenant even with valid IDs; decision logs.</p>
<p><b>Why It Matters:</b> Blocks IDOR/BOLA data leaks and privilege escalation.</p>
<p><b>Technical Breakdown:</b></p>
<ol>

<li>OPA/Cedar policies with unit tests; deny by default.</li>
<li>Log failed authZ decisions to SIEM.</li>
</ol>

<summary>API Security → Secrets Handling</summary>
<p class="tags"><b>Tags:</b> Secrets, Rotation, CI/CD</p>
<p><b>Question:</b> Are secrets avoided in URLs and stored only in secrets managers with rotation?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: IA-5, SC-12<br/>- ISO 27001: A.10.1, A.9.2.4<br/>- SOC 2: CC6.1<br/>- CIS Controls: 3.6</p>
<p><b>Applicability:</b> API clients/backends; CI/CD pipelines.</p>
<p><b>Expected Result:</b> No secrets in GET params/logs; use Secrets Manager/Key Vault/OCI Vault; rotate ≤ 90d.</p>
<p><b>Why It Matters:</b> URL logging leaks secrets; static keys fuel takeover.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Use Authorization headers; redact secrets in telemetry.</li>
<li>Automate rotation; scope secrets per environment.</li>
<li>Prefer OAuth/OIDC or signed requests.</li>
</ol>

<summary>API Security → Replay &amp; Idempotency</summary>
<p class="tags"><b>Tags:</b> Replay, HMAC, Payments</p>
<p><b>Question:</b> Do write endpoints enforce idempotency keys and anti-replay (nonce/timestamp) in signatures?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: SC-23, SC-8(1)<br/>- ISO 27001: A.12.1.2<br/>- SOC 2: CC6.6</p>
<p><b>Applicability:</b> Payments/provisioning/state-changing APIs.</p>
<p><b>Expected Result:</b> Idempotency-Key header; dedup server-side; HMAC covers method+path+body+timestamp; skew ≤ 300s.</p>
<p><b>Why It Matters:</b> Prevents double-spend and signature replay.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Persist idempotency key+hash for ~24h.</li>
<li>Reject stale timestamps; rotate signing keys.</li>
<li>Log replay attempts to SIEM.</li>
</ol>

<summary>API Security (AWS API Gateway) → AuthZ Policy Enforcement</summary>
<p class="tags"><b>Tags:</b> Gateway, AuthZ</p>
<p><b>Question:</b> Is authorization enforced at the gateway with per-route scopes/claims and deny-by-default?</p>
<p><b>Applicable Requirements:</b><br/>- NIST 800-53: AC-3, AC-6<br/>- SOC 2: CC6.6<br/>- NIST CSF: PR.AC</p>
<p><b>Applicability:</b> Public and partner APIs fronted by a gateway.</p>
<p><b>Expected Result:</b> Routes map to required scopes/claims; explicit deny for unspecified routes.</p>
<p><b>Why It Matters:</b> Centralized policy reduces drift and missed checks.</p>
<p><b>Technical Breakdown:</b></p>
<ol>
<li>Map routes→scopes; verify 'sub','aud','scp' claims.</li>
<li>Deny unmatched paths; emit decision logs.</li>
</ol>
